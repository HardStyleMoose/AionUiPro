Purpose
-------
This file is the authoritative Codex handoff for the AionUiPro implementation. It replaces older partial notes. Codex must read this before editing code.

Product goal
------------
AionUiPro should become a polished desktop AI operations control center that keeps the existing AionUI/Gemini CLI experience intact while adding:
- Discord bot control inside Settings -> Remote -> Discord.
- Dual Hermes Docker runtimes: private work agent + public/community Gemma4 bot.
- Obsidian-compatible memory surfaces with strict work/community separation.
- Profile-governed local/cloud model routing.
- Manual llama.cpp Vulkan support for the owner machine.
- Optional Docker Model Runner backend for compatible profiles.
- Fork-safe updates from HardStyleMoose/AionUiPro.
- One-click Start Everything and Save & Exit All workflows.

Non-negotiable constraints
--------------------------
- Do not destructively refactor the app.
- Do not replace Bun, Electron, electron-vite, or electron-builder.
- Do not rename broad renderer/process/shared roots.
- Do not remove existing AionUI functionality.
- Do not change the user’s selected models.
- Do not silently mutate tune profiles or runtime flags.
- Do not silently fallback from Vulkan to CUDA or Docker Model Runner.
- Do not mount private work paths into community runtimes.
- Do not commit secrets or generated runtime data.

Owner machine rule
------------------
Owner profile is Windows + RTX 2060 Super 8 GB. The owner default llama.cpp backend must be manual Vulkan, not CUDA.

Required owner behavior:
- Default backend: manual-vulkan.
- CUDA may exist for other users, but not as silent fallback for the owner profile.
- Docker Model Runner is optional only; it must not replace owner manual Vulkan on Windows.
- If Vulkan validation fails, block launch with a clear error.
- Owner fallback list defaults to empty.

Runtime profile model
---------------------
All model starts must use a saved tune profile.

Core fields:
- id
- name
- scope: private | community | shared | imported
- status: active | last_known_good | candidate | experimental | broken | archived
- ownerDefault: boolean
- backend: manual-vulkan | docker-model-runner | manual-cpu | remote-openai-compatible
- requireBackend: vulkan | cpu | none
- allowFallbackBackends: explicit list only
- binaryPath
- modelPath
- modelAlias
- port
- ctxSize
- gpuLayers
- batchSize
- ubatchSize
- threads
- temperature
- topP
- topK
- repeatPenalty
- maxTokens
- extraArgs
- resource policy
- observations: tokens/sec, RAM, VRAM, last benchmark, stability notes

Rules:
- No saved profile = no launch.
- Missing model path = block launch.
- Incompatible backend = block launch.
- Candidate profiles are never auto-enabled.
- Profile evolution requires user approval.

Vulkan detection implementation
-------------------------------
The owner manual Vulkan profile must validate before launch.

Validation checklist:
1. Windows platform check.
2. GPU inventory from Win32_VideoController.
3. Vulkan runtime via vulkaninfo.exe or vulkaninfoSDK.exe.
4. Vulkan physical device enumeration.
5. llama-server.exe exists.
6. llama-cli.exe smoke test if available.
7. GGUF model path exists.
8. Profile backend is manual-vulkan.
9. If any required step fails, block launch.

Required error messages:
- VULKAN_RUNTIME_MISSING: Vulkan is required by this tune profile, but the Windows Vulkan runtime was not detected.
- VULKAN_DEVICE_NOT_FOUND: A Vulkan runtime exists, but no usable Vulkan physical device was reported.
- LLAMA_VULKAN_BINARY_MISSING: No Vulkan-capable llama.cpp server binary was found.
- LLAMA_VULKAN_SMOKETEST_FAILED: The configured llama.cpp binary exists, but the Vulkan smoke test did not initialize the Vulkan backend.
- DMR_WINDOWS_NOT_VULKAN: Docker Model Runner is available, but on Windows it cannot satisfy this Vulkan-pinned profile.
- PROFILE_FALLBACK_BLOCKED: Automatic fallback is disabled for this owner profile.

Docker Model Runner policy
--------------------------
Docker Model Runner is optional and profile-selected.

Rules:
- Keep manual Vulkan as owner default.
- DMR must not replace manual Vulkan on Windows for owner profile.
- DMR may be used by non-Vulkan or non-owner profiles.
- DMR profiles keep settings in AionUiPro by default.
- Do not globally mutate DMR model config unless user explicitly requests it.
- Probe DMR at localhost:12434/engines/v1/models or the configured endpoint.

Obsidian integration recommendation
-----------------------------------
Use AionUiPro as the primary integration and Obsidian as a memory surface.

Recommended architecture:
- AionUiPro owns runtime control, secrets, profile rules, Discord routing, Docker lifecycle, and memory boundaries.
- Obsidian stores safe markdown summaries, playbooks, public/community notes, and reviewed skill candidates.
- The Obsidian Agent Client plugin can be supported as an optional bridge, but it should not become the primary runtime controller.

Why:
- AionUiPro can enforce secrets, backend rules, and work/community separation.
- Obsidian plugins are useful for human-readable notes but weaker for process supervision and security boundaries.
- A plugin bridge is useful for manual agent interaction inside the vault, not for controlling Docker, Discord, or Vulkan launch policy.

Obsidian rules:
- no secrets in vaults
- no API keys in markdown
- separate work and community vaults
- community notes cannot auto-promote into work memory
- owner approval required for promotion

Save & Exit All
---------------
UI action: About -> Save & Exit All
Script: scripts/save-and-exit-all.ps1

Flow:
1. Stop accepting new Discord jobs.
2. Flush active sessions.
3. Save/summarize memory where supported.
4. Write Obsidian summaries if enabled.
5. Stop owner Vulkan llama.cpp process if app-launched.
6. Stop Hermes Work Agent.
7. Stop Gemma4 Community Bot unless keep-alive selected.
8. Stop Discord bridge.
9. Release app-owned ports.
10. Call Electron app.quit() from the main process.

Important:
- never delete Docker volumes during Save & Exit All
- never require manual port killing
- shutdown must be repeatable and visible
