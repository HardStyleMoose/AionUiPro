Dual Hermes runtime design:
---------------------------
There must be two separated Hermes runtimes.

1) Hermes Work Agent
   - Container name: hermes-work
   - Default host port: 8642 -> container port 8642
   - Data path: C:\AI\HermesDocker\work-data
   - Memory scope: private
   - Skills scope: private
   - Used by AionUiPro Desktop by default for the user’s private work.
   - Used by Discord only through owner/admin-only /work commands.
   - May access project workspaces only when explicitly configured.
   - File/command execution requires approval.
   - Should be on-demand by default to save resources.

2) Gemma4 Community Discord Chatbot
   - Container name: hermes-gemma4-community
   - Default host port: 8643 -> container port 8642
   - Data path: C:\AI\HermesDocker\gemma4-community-data
   - Memory scope: community only
   - Skills scope: community only
   - Used by public Discord /ask commands and public mentions.
   - Must never access private work memory.
   - Must never access private Obsidian work vault.
   - Must never access private repo mounts.
   - Must never access host execution tools by default.
   - May self-improve by creating community skill candidates, but those candidates must be reviewed before promotion.
   - Should be the always-on runtime if Discord community bot is enabled.

Memory isolation rules:
-----------------------
- Work memory and community memory are separate.
- Work skills and community skills are separate.
- Community learned memories cannot automatically enter the work agent.
- Community learned skills cannot automatically enter the work agent.
- Promotion requires explicit owner approval.
- Promotion flow supports: approve, reject, redact then approve, keep community-only.
- Obsidian is for safe human-readable notes, playbooks, and memory summaries, not secrets.
- Secrets are stored in encrypted app storage or OS-safe storage, not Obsidian and not Git.

Docker compose template to generate:
------------------------------------
services:
  hermes-work:
    image: nousresearch/hermes-agent:latest
    container_name: hermes-work
    restart: "no"
    profiles: ["work"]
    volumes:
      - ../work-data:/opt/data
      - ${AIONUI_WORK_OBSIDIAN_VAULT:-C:/AI/HermesDocker/work-obsidian}:/opt/obsidian:rw
    ports:
      - "8642:8642"
    environment:
      HERMES_HOME: /opt/data
      HERMES_PROFILE: work
      HERMES_DASHBOARD: "1"
    command: gateway run

  hermes-gemma4-community:
    image: nousresearch/hermes-agent:latest
    container_name: hermes-gemma4-community
    restart: unless-stopped
    profiles: ["community"]
    volumes:
      - ../gemma4-community-data:/opt/data
      - ${AIONUI_COMMUNITY_OBSIDIAN_VAULT:-C:/AI/HermesDocker/community-obsidian}:/opt/obsidian:rw
    ports:
      - "8643:8642"
    environment:
      HERMES_HOME: /opt/data
      HERMES_PROFILE: gemma4-community
      HERMES_DASHBOARD: "1"
    command: gateway run

Start Everything workflow:
--------------------------
Add an app service/action and scripts.

UI action: Start Everything
Script: scripts/start-everything.ps1

The flow should:
1. Check Docker CLI availability.
2. Check Docker Desktop/engine running.
3. Start Docker Desktop if possible, otherwise show clear instructions.
4. Generate docker-compose.yml if missing.
5. Start hermes-gemma4-community.
6. Optionally start hermes-work depending on setting.
7. Wait for health checks:
   - community gateway: http://127.0.0.1:8643
   - work gateway: http://127.0.0.1:8642
8. Start Discord bridge if enabled.
9. Sync slash commands if configured.
10. Show final status summary in AionUI.

Save & Exit All workflow:
-------------------------
Add an app service/action and scripts.

UI action: Save & Exit All
Script: scripts/save-and-exit-all.ps1

The flow should:
1. Stop accepting new Discord work.
2. Flush active sessions.
3. Save/summarize memory where supported.
4. Write Obsidian summaries if enabled.
5. Stop Hermes Work Agent.
6. Stop Gemma4 Community Bot only if full shutdown is selected; otherwise leave community bot running if configured.
7. Stop Discord bridge.
8. Stop local background services.
9. Release app-owned ports.
10. Call Electron app.quit() directly so close-to-tray behavior does not keep services running.

Important workflow requirement:
- The user does not want to manually close ports or kill terminals.
- The shutdown must be safe, visible, and repeatable.
